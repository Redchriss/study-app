## Backend Access Control Model

### 1. User Types & Role Hierarchy

**Custom User model** (`apps/accounts/models.py`):
- `User(AbstractUser)` — base user with `phone` field, `is_parent` property (true if `ParentChildLink` exists)
- `UserProfile` — one-to-one profile holding all role-specific data

| User Type | Identification | Key profile fields |
|---|---|---|
| **Primary Student** | `profile.education_level='primary'` | `standard` (1-8), `term`, `primary_school` |
| **Secondary Student** | `profile.education_level='secondary'` | `form` (1-4), `term`, `secondary_school` |
| **Tertiary Student** | `profile.education_level='tertiary'` | `university`, `program` |
| **Adult/Parent** | `user.is_parent == True` | linked to children via `ParentChildLink` |
| **Admin** | `user.is_staff == True` | Django admin access (`/admin/`) |
| **Moderator** | `Membership.role == 'moderator'` | Community-level, per-community |
| **Community Admin** | `Membership.role == 'admin'` | Community creator or appointed admin |
| **Kid (child)** | `profile.is_child == True` | `pin_code`, `parent` FK, `child_name`, `child_education_track` |
| **Anonymous** | `user.is_anonymous` | Register/login required |

**Role hierarchy:** `Admin (is_staff)` > `Community Admin` > `Community Moderator` > `Member/Parent/Student/Kid` > `Anonymous`

### 2. Education Levels

**Modeled in `UserProfile.education_level`** with choices: `primary`, `secondary`, `tertiary`.

| Level | School FK | Subject FK | Grade field | Exam references |
|---|---|---|---|---|
| **Primary** | `PrimarySchool` (Malawi schools) | `PrimarySubject` (infant/junior/senior) | `standard` (1-8) | `is_pslce_subject` |
| **Secondary** | `SecondarySchool` (public/private/mission/CDSS) | `SecondarySubject` (core/science/arts/technical) | `form` (1-4) | `is_jce_subject`, `is_msce_subject` |
| **Tertiary** | n/a (uses `University`) | `Subject` with `program` FK | n/a (uses `program`) | n/a |

**Access gating by education level (backend GraphQL):**
- **Materials**: `_user_can_access_material()` checks `profile.education_level` + grade against subject. Tertiary students scoped to `program_id`.
- **Quizzes**: filters by `subject__education_level=profile.education_level` for non-tertiary.
- **Past Papers**: available only for primary and secondary — returns `[]` for tertiary.
- **Education query**: reference data — no auth gating.

### 3. Kids Mode

**Architecture:** Kids have their own `User` account linked to a parent via `ParentChildLink`.

**Child account creation:** Parent calls `create_child_profile` mutation → creates `User` + `UserProfile(is_child=True, pin_code=4-digit, parent=parent_profile)` + `ParentChildLink`.

**Kid login:** `KidLogin` mutation validates username + 4-digit PIN.

**Kid data models** (`apps/kids/models.py`):
- `KidsLesson` — cached AI-generated lessons (chunks_json for emoji+text display)
- `KidProgress` — per-subject progress
- `KidLessonState` — per-topic mastery with spaced repetition
- `KidRewardProfile` — XP, level, coins, companions (Sprout/Rocket/Lion/Owl)
- `KidBadgeAward` — achievements (First Spark, Weekly Wonder, Topic Tamer, etc.)
- `KidDailySnapshot` — daily engagement rings + chest

### 4. Communities & Moderation

**Community roles** (defined in `Membership` model):

| Role | Permissions |
|---|---|
| `member` | Browse, post (subject to karma/age gates), comment, vote |
| `moderator` | Remove posts/comments, ban/mute, pin/lock, view reports + mod log, access modmail |
| `admin` | All mod actions + edit community settings, sidebar, add/remove moderators |
| Banned (`is_banned`) | Cannot post or comment |

**Permission decorators** (`apps/communities/permissions.py`):
- `require_community_member` — raises `PermissionError` if not member
- `require_community_mod` — checks `role in ['moderator', 'admin']`
- `require_community_admin` — checks `role == 'admin'`
- `check_not_banned` — checks `is_banned` and `ban_expires_at`
- `check_karma_gate` — checks `profile.total_karma >= community.min_karma_to_post`
- `check_account_age` — checks `account.age >= community.min_account_age_days`

**Gates applied during post creation:** `check_all_post_gates()` calls all three guards.

### 5. Subscription/Paywall

| Pattern | Description | Examples |
|---|---|---|
| `@login_required` decorator | Returns `PermissionError` if anonymous | Most mutations |
| Anonymous check in resolver | Returns `None`/`[]` if anonymous | `me`, `notifications` |

## Summary of Key Findings

1. **3 missing routes**: `/materials`, `/materials/:slug`, `/materials/:slug/read` are navigated via `context.push()` but have NO GoRoute entry. These screens work because Flutter's `context.push()` resolves relative paths, but they won't respond to deep links or page refresh properly.

2. **No education-level gating on client**: Backend filters materials/quizzes/pastpapers by education level, but the Flutter client does not check or display education-level filtering. `educationLevel` is returned in GraphQL responses but unused on the client for gating.

3. **No role-based routing on Flutter side**: The router redirect only checks `isAuthenticated` and `onboardingComplete`. There is no admin, mod, or adult/parent role separation in screen access on the client side. Community role branching happens within screens (conditionally rendered widgets), not at the route level.

4. **All screens gated by central redirect**: No individual screen performs its own auth check. The GoRouter `redirect` callback handles all auth gating.

5. **Kids mode bypasses main auth**: `/kids/*` routes are excluded from the main auth redirect. Kids use a separate `kidAuthStateProvider` with PIN-based login. This is correct architecture.

6. **AI credits not checked on client**: The client sends AI action requests without checking the user's credit balance. If credit is 0, the backend rejects with an error. The `/upgrade` screen shows credit info but there's no pre-flight check before AI actions.

7. **Premium content not enforced**: `StudyMaterial.is_premium` and `Quiz.is_premium` are defined on models and returned in GraphQL, but neither the backend resolvers nor the client enforce premium gating. Currently only the AI credits system is actively enforced.

8. **Parent dashboard gap**: `ParentKidsProgressScreen` uses the main `graphqlClientProvider` (parent auth) but its GraphQL queries should be checked to ensure they use the correct auth token. Kid screens correctly use a separate `KidGraphqlClient`.

9. **Mod panel has no client-side guard**: Any authenticated user can navigate to `/y/:slug/mod` directly — there is no client-side route-level guard. The only protection is that the shield icon (entry point) only renders for mods, and the backend returns no data for non-mods.

10. **Bottom nav has 4 tabs**: Home, Discover, Inbox, Profile. Materials, quizzes, scanner, and AI tutor are accessible only via navigation from within screens (dashboard, profile settings, etc.).
